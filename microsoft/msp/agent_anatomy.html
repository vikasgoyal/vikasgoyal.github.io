Architecture Reference

Agent Anatomy

A reference model for reading an enterprise agent runtime from left to right: ingress, context assembly, LLM reasoning, orchestration, and egress, with the model acting as the inference core and memory, integration surfaces, and control-plane services shaping how it operates.

Start with the diagram, then use the node reference to inspect each block in the same order the runtime executes.

Node Reference

Diagram Nodes And Runtime Contracts

These components define what executes inside the loop, what is delegated outward, and what remains under platform control.

Ingress And Triggers

Captures user requests, app events, identities, and upstream signals that enter the runtime boundary.

Instruction Stack

System prompts, policy, schemas, tool contracts, and escalation rules define the runtime envelope before execution starts.

Context Assembly

Builds the effective prompt from retrieval results, session state, memory lookups, identity, and runtime metadata.

LLM Reasoning Core

Runs model inference over the assembled context, then performs decomposition, ranking, routing, and step selection across candidate actions and agents.

External Tools

Represents the callable surfaces outside the agent boundary, including MCP servers, A2A peers, APIs, and workflow endpoints.

Orchestration Surface

Chooses when to invoke external tools, coordinates calls, and normalizes returned state for the next runtime step.

Memory Subsystem

Stores durable user facts, summaries, checkpoints, and retrieval keys for cross-session continuity.

Response And Egress

Materializes structured JSON, UI payloads, citations, messages, and side-effect metadata.

Guardrails And Policy

Enforces safety policy, data residency, authZ boundaries, approval gates, and fallback behavior.

Observability And Eval

Captures traces, tokens, latency, tool outcomes, eval runs, and regression signals.

Execution Path

Five-Step Runtime Path

01

Enter

Accept a user request, system event, or trigger and establish identity and policy context.

02

Assemble Context

Resolve retrieval scope, memory lookups, prompt instructions, and runtime metadata before inference.

03

Reason

Interpret intent, rank candidate actions, and decide whether the next step stays in-model or leaves the boundary.

04

Orchestrate

Invoke tools, MCP sessions, A2A peers, or workflows and normalize their outputs back into runtime state.

05

Respond + Record

Return answers, persist memory updates, emit output contracts, and capture telemetry for governance and evaluation.

Architecture Principles

Production Design Constraints

Production agent platforms depend on explicit contracts between inference, context, orchestration, and control-plane services.

Separation Of Runtime Concerns

Keep prompt assembly, reasoning, orchestration, memory, and response shaping independently observable.

Contracts Before Calls

Tooling should be schema-bound and policy-checked across MCP, A2A, or direct API surfaces.

Traceability Over Intuition

Favor trace logs, eval runs, latency budgets, and regression signals over prompt-only reasoning.
